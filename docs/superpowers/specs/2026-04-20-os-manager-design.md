# 操作系统智能代理 Web 方案设计

## 目标

构建一个面向 Linux 服务器管理场景的 Web 版操作系统智能代理，覆盖基础运维、跨发行版适配、装机初始化和任务透明化执行四类能力，并在真实命令执行前后提供风险提示、副作用说明和回滚建议。

## 设计结论

采用 Web 前端 + Node.js/TypeScript 后端 + 任务管理器 + 发行版适配层的方案。

原因如下：

1. Web 页面更符合“去命令行化”的比赛导向，演示效果明显优于纯 CLI。
2. 任务管理器可以把一次自然语言请求拆成多个可观察步骤，直接解决“执行过程黑盒不透明”的痛点。
3. 发行版适配层可以把 openEuler、CentOS、Ubuntu 的差异封装掉，避免命令选择逻辑散落在 UI 或执行器中。

## 核心能力范围

首版聚焦以下能力：

1. 环境感知：识别发行版、包管理器、Python 包管理器、是否支持 systemd。
2. 基础运维：磁盘使用情况、文件检索、进程/端口查询、普通用户创建与删除。
3. 初始化任务：配置 yum/dnf/apt 镜像源、配置 pip 源、进行常用环境检查。
4. 透明执行：显示任务拆解、命令选择依据、潜在副作用、失败定位、回滚建议。
5. 风险控制：对删用户、改源、改系统文件等动作打风险标签，并要求二次确认。

## 创新点

1. 跨发行版命令决策引擎：同一意图在不同发行版下生成不同执行计划，并给出“为什么选这条命令”。
2. 可解释任务管理器：参考 multica 的任务编排展示方式，把请求拆成任务树、步骤日志和状态时间线。
3. 安全与回滚顾问：在执行前提示副作用，在执行后给出可复制的回滚命令或人工恢复建议。

## 架构

### 1. Web 前端

职责：

1. 输入自然语言任务。
2. 展示环境信息卡片。
3. 展示任务计划、步骤状态、风险提示和执行输出。
4. 在高风险场景中触发二次确认。

### 2. API 服务

职责：

1. 接收用户输入。
2. 调用意图解析器、发行版检测器和计划生成器。
3. 返回任务计划预览。
4. 在用户确认后调用执行器运行命令。

### 3. 意图解析器

职责：

1. 将自然语言映射为标准化任务类型。
2. 提取参数，例如用户名、文件名、端口号、镜像源目标。
3. 评估风险等级。

首版采用规则 + 关键词方式，保留后续接入大模型的扩展点。

### 4. 发行版适配层

职责：

1. 识别 openEuler、CentOS、Ubuntu。
2. 提供包管理器、服务管理器、默认命令优先级等能力画像。
3. 为计划生成器输出适合当前环境的命令模板。

### 5. 任务计划生成器

职责：

1. 将标准任务映射为多步执行计划。
2. 为每一步附带 explanation、sideEffects、rollbackHint。
3. 对高风险步骤标记 requiresConfirmation。

### 6. 命令执行器

职责：

1. 串行执行计划步骤。
2. 采集 stdout、stderr、exitCode。
3. 失败时中断并记录定位信息。
4. 支持 dry-run 模式，便于本地演示和开发。

## 数据流

1. 用户在 Web 页面输入请求。
2. 后端解析意图并识别当前发行版。
3. 计划生成器输出任务树。
4. 前端展示任务预览和风险说明。
5. 用户确认后开始执行。
6. 执行器逐步返回日志和结果。
7. 前端展示总结、失败定位和回滚建议。

## 错误处理

1. 发行版无法识别时，降级为 generic-linux 配置，只开放查询类能力。
2. 参数缺失时，不直接执行，返回补充信息请求。
3. 高风险任务未确认时，不允许执行。
4. 命令失败时，返回失败步骤、原始错误输出和下一步建议。

## 测试策略

1. 单元测试：发行版识别、意图解析、任务规划、风险标记。
2. 组件测试：前端任务时间线和风险确认交互。
3. 集成测试：API 从输入到计划生成的闭环。
4. 手工验证：在 Linux 环境实际演示磁盘查询、文件检索、用户管理和镜像源配置。

## 首版交付边界

首版不做语音输入，不接入 SSH 远控，不做复杂多 Agent 协同。重点是把“能真实执行、跨发行版适配、过程可解释”这三件事做扎实。
