1. 背景说明
当前主流服务器操作系统（如openEuler、 CentOS、 Ubuntu Server 等）仍以命令行（CLI）
作为主要交互手段。但在实际应用中存在以下痛点：
• 操作门槛高：用户在完成系统安装并登录后，往往缺乏明确的操作指引；
• 环境差异大：不同 Linux 发行版间的工具链、参数配置及最佳实践存在显著差异；
• 记忆成本重：实现高效的系统管理通常要求用户熟练掌握大量指令。
综上所述，系统使用的核心障碍并非安装部署，而是人机之间过高的“沟通成本”
。
随着大模型与 AI Agent 技术的发展，新型交互范式应运而生：用户可通过自然语言描
述需求，由 AI 解析意图并代为执行相应的服务器操作。例如：
•
“查询当前磁盘剩余空间”
•
“创建标准用户账号”
AI 将根据当前操作系统的实际环境，调用该系统推荐的最佳实践路径，并以自然语言
向用户反馈执行过程与结果。
在本题目中，承担此类职能的 AI Agent 统称为 “操作系统智能代理”（以下简称
“智能代理”）。该智能代理既可直接部署于目标服务器，亦可通过远程协议（如 SSH）进
行跨环境操作。
对用户而言，智能代理的具体部署位置是透明的，用户仅需通过自然语言下达指令。
基于此，一个深层课题随之出现：当 AI 能够理解并代理复杂的系统操作时，服务器操
作系统是否仍须绑定命令行交互模式？
本题目旨在响应这一现实需求，鼓励参赛者开发真实、可落地的操作系统智能代理，积
极探索 AI 作为服务器操作系统交互界面的创新形态。
2. 任务目标
本题目旨在引导参赛者构建具备实际应用价值的操作系统智能代理。
无论采用本地部署或远程连接（如 SSH）模式，智能代理均应具备代表用户对指定 Linux
服务器执行管理操作的能力。用户通过自然语言（文本或语音）输入需求后，智能代理需准
确识别意图，结合环境自动化完成操作，并对执行逻辑与结果进行清晰表述。
评分将基于参赛作品在各维度的综合表现。
在满足基础功能的前提下，作品在安全意识、风险控制、连续任务处理及交互形态等方

2.1. 基础能力要求
实现支持自然语言驱动的服务器操作系统智能代理，并满足以下准则：
2.1.1. 提供可运行或可交互的工具实现
形态包括但不限于 CLI、TUI、客户端或 Web 端；
2.1.2. 支持基础系统管理需求的自然语言处理
包括但不限于：
• 磁盘使用情况监测
• 文件或目录检索
• 进程及端口状态查询
• 普通用户的创建与删除
2.1.3. 智能代理必备能力
• 准确的意图解析能力；
• 操作系统层面的任务执行能力；
• 过程与结果的自然语言反馈能力。
2.2. 进阶能力要求（安全意识与风险控制）
在具备基础能力的基础上，鼓励作品体现系统管理场景下的安全意识与风控机制：
2.2.1. 高风险或敏感操作识别
包括但不限于以下维度：
• 系统核心文件或目录的删除；
• 关键安全配置的篡改；
• 大范围的用户权限变更。

2.2.2. 风险预警
包括但不限于以下维度：
• 风险预警及二次确认请求；
• 操作范围的限制与截断；
• 拒绝不合理或非法的高风险指令。
2.2.3. 行为可解释
针对安全判定逻辑与处置结果提供清晰、可理解的解释。
2.3. 探索类能力方向（交互形态与使用方式探索）
鼓励参赛队伍探索“AI 作为操作系统主要交互入口”的前瞻形态，包括但不限于：
• 支持语音输入、语音对话等多模态交互；
• 具备多轮对话能力，能够精准理解并利用上下文信息；
• 实现“去命令行化”的系统管理体验；
• 支撑多步连续任务的编排处理，并进行统一的反馈。
3. 输入说明

• 参赛队伍需自行搭建 Linux 服务器操作系统环境（如 openEuler、 CentOS、Ubuntu
Server 等） 。
• 参赛队伍根据方案需求，自行配置运行环境及软件依赖。
• 智能代理既可本地部署，亦支持在外部环境（如 Windows）中通过远程连接代理操
作服务器。
